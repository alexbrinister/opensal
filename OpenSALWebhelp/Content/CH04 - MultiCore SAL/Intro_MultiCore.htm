Using MultiCore SAL

Overview

MCSAL was developed to help you migrate your single-core (serial) SAL applications to multi-core (parallel) processing environments, without having to change every SAL call in your programs.

MCSAL includes macros that determine whether to run a single-core or multi-core version of a function. Therefore, your applications can contain both single-core and multi-core SAL calls.

You can code your 32-bit applications using C or C++.

MCSAL Libraries

MCSAL has a performance library and a development library. The development library provides TATL™ trace events for each MCSAL function. For more information on TATL trace events, see the Trace Analysis Tool and Library (TATL) User Guide or the MultiCore Plus Open Development Suite User Guide.

Multi-Core Thread Execution

MCSAL is used from one process thread only, the manager thread. This thread creates and manages one or more worker threads. The worker and manager threads share the work performed by a SAL function.

Figure 12-1: Multi-Core Thread Execution

Mercury recommends that you do not use more than one manager thread. If you do use more than one manager thread, and they both need to invoke MCSAL functions, they must not do so at the same time. Simultaneous invocation of any MCSAL function by more than one thread can result in undefined behavior.

However, if there is one thread making single-core SAL calls (see SAL_SINGLE ) and one thread making multi-core calls, synchronization between the threads is unnecessary.

When determining how to divide processing among the manager and the worker threads, MCSAL assumes the manager and workers have equal and exclusive access to the cores, and are not competing with other threads for processing resources.
